# HEM-CM 系统架构

HEM-CM 是一个基于 Next.js App Router、Prisma 与 SQLite 构建的本地优先事件智能研究终端。这个仓库把产品界面、可解释事件引擎、回放与验证流程，以及面向开源分发的仓库级文档素材，统一组织在同一个 TypeScript 代码库中。

## 系统概览

从整体上看，系统会把外部信源输入转换成结构化事件对象，通过 Prisma 持久化到 SQLite，再把这些结果渲染到首页、事件详情、市场映射、信号中心、回测实验室和模型管理等产品界面中。

这个架构是有明确倾向的：

- 以产品工作流为中心，而不是以零散 notebook 为中心
- 以可解释性为中心，而不是黑盒输出
- 以模块分层为中心，而不是把业务逻辑塞进页面
- 以本地优先为中心，让完整流程可以在单机上被审查和理解

## 运行层次

### 1. Web 应用外壳

App Router 的入口位于 `app/`。根布局负责初始化全局能力：

- 主题提供器
- 语言解析与字典注入
- 导航壳层
- 服务端调度器启动

这样可以把全局状态和布局职责从单个页面中抽离出来。

### 2. 服务端数据组装层

`lib/data.ts` 是面向页面的数据聚合层，负责：

- 确保演示数据已初始化
- 读取运行时偏好
- 通过 Prisma 读取事件、信号和市场映射数据
- 应用 demo mode 的运行时修正
- 把持久化数据整理成页面直接可消费的视图模型

因此页面本身可以保持相对轻量，读侧组装职责集中在这一层。

### 3. 事件与模型引擎

领域引擎位于 `modules/`。主事件流水线会组合多个专门引擎：

- 分类引擎
- 阶段引擎
- 信号引擎
- 因果引擎
- 预测引擎
- 市场映射引擎
- 回测引擎

`modules/event-engine/index.ts` 是编排入口，负责把一个原始事件样本转换成完整事件包。

### 4. 持久化层

Prisma 模型定义了系统的长期结构：

- 事件与事件簇
- 信号
- 路径预测
- 市场与行业影响
- 回测结果
- 模型配置与信源配置
- 用户偏好与预警规则

SQLite 是默认本地数据库，这让安装和调试都更轻量，也更方便理解整个栈。

## 关键目录

### `app/`

承载 Next.js 的页面路由与 API 路由。

- 包括首页、事件详情、信号、市场、回测、设置、模型管理等页面
- `app/api/` 下则是数据访问与产品动作入口

### `components/`

承载按产品界面拆分的可复用 UI 组件，以及共享布局组件。

### `lib/`

承载应用服务与运行时边界。

关键职责包括：

- Prisma 数据访问
- locale 与字典处理
- 首页与详情页数据组装
- 调度器启动
- 公共工具与类型定义

### `modules/`

承载可解释领域引擎。每个模块只负责事件推理的一部分，避免把全部逻辑混入页面或路由处理器中。

### `prisma/`

承载数据库 schema、初始化 SQL 和 seed 逻辑。

### `docs/`

承载路线图、社区、架构、GitHub 展示与发布相关文档，是仓库对外说明层。

## 主要数据流

### 1. 信源接入

原始信源既可以来自 seed 的演示数据，也可以来自实时信源接入流程。Source adapter 会把异构外部输入标准化成仓库内部能消费的样本。

### 2. 事件包构建

事件引擎会把原始样本转换成结构化输出：

- 事件家族分类
- 当前阶段识别
- 信号聚合指标
- 母模型状态
- 因果链
- 多时间窗路径预测
- 市场与行业映射
- 回放或回测对象

### 3. 持久化

这些结构化结果会通过 Prisma 写入数据库，这样多个产品界面可以复用，而不必每次都重新计算。

### 4. 读侧整形

`lib/data.ts` 会把持久化对象进一步整理成首页卡片、热力图、市场表格、事件详情和信号流等 UI 数据结构。

### 5. 产品渲染

最后由 Next.js 页面在 App Shell 中完成渲染，并使用 locale 相关标签与格式化能力输出给用户。

## 核心领域模型

这个仓库的中心设计是：一个可复用的通用因果母模型，加上六类可配置的事件子模型。

### 母模型

母模型负责表达稳定的推理原语，例如：

- 行为体
- 驱动因素
- 约束因素
- 触发条件
- 响应方式
- 传导路径

### 事件家族专化

每个事件家族在母模型之上增加自己的分类逻辑、阶段偏置、路径模板和市场映射方式。这样既能保持统一推理语言，又能支持不同主题的专门化建模。

### 多时间窗预测

系统会对固定时间窗生成预测：

- 7 天
- 30 天
- 90 天
- 180 天
- 365 天

每个时间窗下可以存在多条路径，并带有成立条件与失效信号，因此系统比单一风险分数看板更容易审查。

## 持久化模型

Prisma schema 按职责拆分写模型：

- `EventCluster` 用于聚合相关事件对象
- `Event` 保存当前规范化事件状态
- `Signal` 保存源级证据
- `PathPrediction` 保存结构化情景路径
- `MarketImpact` 与 `SectorImpact` 保存下游市场视角
- `BacktestRun` 保存回放与验证结果
- `ModelConfig` 与 `SourceConfig` 保存可配置行为
- `UserPreference` 保存产品级偏好，例如语言与刷新间隔

这个结构同时支持产品渲染与后续模型迭代。

## 运行时与安全边界

仓库中有一些刻意维持的边界：

- Prisma 被保护，避免误进入浏览器端图谱
- 调度器代码严格限定在服务端
- locale 在服务端解析，再通过 client-safe provider 注入
- AppShell 改为纯 props 驱动，防止 server-only 逻辑泄漏进 client graph

这些边界之所以重要，是因为仓库同时混用了 Server Components、Client Components 与本地运行时服务。

## 调度器与刷新行为

调度器在根布局启动，目前以 cron 周期刷新运行时元数据。这样可以把周期性任务集中管理，而不是绑定在页面请求上。

对于本地优先开发来说，这个设计是刻意保持轻量的：

- 不依赖外部队列
- 不需要独立 worker 服务
- 不引入重型基础设施编排

## 验证理念

HEM-CM 把验证视为产品架构的一部分，而不是事后补的脚本。

关键验证面包括：

- 首次运行可见的演示数据
- `npm run build` 的构建验证
- `npm run validate` 的回放与规则验证
- 产品内部可直接查看的回测界面

这让仓库更容易演示、审查和扩展。

## 扩展点

当前最重要的扩展方向包括：

- 新的 source adapters
- 新的事件家族模板
- 更强的阶段与路径逻辑
- 更丰富的 replay 指标
- 更强的 explainability 面板
- 更好的导出能力与 AI-ready 结构化输出

## 架构总结

HEM-CM 可以理解为一个分层的本地研究系统：

- Next.js 提供产品界面与路由
- `lib/` 负责把运行时数据整形成 UI 可消费格式
- `modules/` 负责可解释事件推理
- Prisma 与 SQLite 负责持久化结构化输出
- 仓库文档与治理文件负责支持开源协作与传播

正是这种分层，使项目既保留了产品感，又足够透明，便于研究、调试和后续 AI 增强。
